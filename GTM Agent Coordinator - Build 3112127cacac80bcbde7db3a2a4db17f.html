1. Project Overview

1.1 Product Summary

The GTM Agent Coordinator is a cloud-native coordination platform that acts as a centralized control point for all AI agents operating within a customer's go-to-market (GTM) stack. It provides identity management, preflight action authorization, record resolution, ownership intelligence, scheduling, audit logging, collision detection, policy enforcement, rate limiting, outcome attribution, and partner integrations — all exposed via REST APIs and the Model Context Protocol (MCP).

1.2 Design Principles

  1. API-first: Every capability is a REST API. The UI (Dashboard) is a consumer of APIs, not a separate system.
  1. Cloud-native: Deployed on AWS. No Salesforce-embedded code in the hot path. SFDC is a data source, not the runtime.
  1. Sub-second latency: The Preflight API (the core control point) must respond in ≤500ms at p95. All architectural decisions serve this constraint.
  1. Immutable audit: Every action request, decision, and outcome is recorded in an append-only ledger. Nothing is deletable.
  1. Multi-tenant from day one: All data is tenant-isolated. Architecture supports horizontal scaling.
  1. Customer-facing quality: Production-grade error handling, input validation, rate limiting, observability, and documentation. This is not an internal tool.
  1. Graceful degradation: If any subsystem fails, the system degrades gracefully (e.g., fail-open mode on Preflight) rather than blocking all agent activity.

1.3 Key Terminology

TermDefinition
AgentAn AI-powered tool that takes autonomous actions on GTM records (e.g., sends emails, books meetings, enriches data)
PreflightThe checkpoint API call an agent makes before taking an action. Returns GO / NO_GO / WAIT / REDIRECT
Canonical RecordThe Coordinator's internal normalized representation of a person/account, resolved from multiple identifiers
Action LedgerThe immutable append-only log of all agent interactions
CollisionTwo or more agents attempting conflicting actions on the same record within a time window
Communication BudgetPer-record limits on outbound touches across all agents within configurable time windows
PolicyA customer-configurable rule that governs agent behavior (sequencing, approval gates, priorities, etc.)

2.4 Multi-Tenancy Model

Every table and cache key is scoped by tenant_id. The tenant is the LeanData customer organization.

  • Database: All tables include a tenant_id column. Row-level security (RLS) policies enforce isolation at the PostgreSQL level.
  • Cache: All Redis keys are prefixed with t:{tenant_id}:.
  • API: tenant_id is derived from the API key during authentication. It is never provided by the caller.
  • OpenSearch: Separate index aliases per tenant, backed by shared indices with tenant_id filtering.

3.2 Redis Cache Schema

# Agent auth cache (avoids DB lookup on every request)
t:{tenant_id}:agent:{api_key_hash} → JSON { agent_id, tenant_id, status, agent_type }
TTL: 300s (5 minutes)

# Record cache (hot records for Preflight)
t:{tenant_id}:record:{canonical_record_id} → JSON { owner, account, opp, last_actions[] }
TTL: 900s (15 minutes, refreshed on sync)

# Record lookup by email
t:{tenant_id}:email:{normalized_email} → canonical_record_id
TTL: 900s

# Recent actions per record (for collision detection)
t:{tenant_id}:actions:{canonical_record_id} → Sorted Set (score = timestamp)
Members: JSON { agent_id, action_type, request_id, timestamp }
TTL: 86400s (24 hours)

# Communication budget counters (sliding window)
t:{tenant_id}:budget:{budget_id}:{canonical_record_id} → Sorted Set (score = timestamp)
Members: request_id
TTL: max(window_duration) + 1 hour

# Preflight idempotency (dedup identical requests)
t:{tenant_id}:preflight_dedup:{agent_id}:{record_id}:{action_type} → JSON response
TTL: 5s

# Agent status cache
t:{tenant_id}:agent_status:{agent_id} → status string
TTL: 60s

API Endpoints

Resolution Logic (Ordered)

  1. Exact email match: Normalize email (lowercase, trim). Lookup in Redis → PostgreSQL.
  1. SFDC ID match: If sfdc_lead_id or sfdc_contact_id provided, direct lookup.
  1. Phone match: Normalize phone (E.164 format). Lookup by normalized phone.
  1. Fuzzy name + company match: If no exact match found, use trigram similarity (pg_trgm extension) on last_name + company_name. Return results with match_confidence score.
  1. LinkedIn URL match: Normalize LinkedIn URL, lookup by exact match.
  1. Create new canonical record: If no match is found and the identifiers are sufficient (at minimum: email OR sfdc_id), create a new canonical record and flag it for SFDC sync enrichment.

Booking Logic (Exactly-Once Guarantee)

  1. Double-check availability: Before booking, re-check the calendar slot is still free (prevent race conditions).
  1. Acquire distributed lock: Redis lock on booking:{host_email}:{slot_start} with 30s TTL.
  1. Create calendar event via Google Calendar API or Microsoft Graph API.
  1. Write to PostgreSQL (scheduled_meetings table).
  1. Write to CRM (Salesforce Event, async via SQS — fire-and-forget with retry).
  1. Write to Ledger (async via SQS).
  1. Release lock.
  1. If any step fails after calendar event creation, compensate (delete calendar event).

Calendar Token Management

Per LeanData's BookIt learnings on token deauthorization:

  • Store OAuth tokens encrypted in PostgreSQL.
  • Implement proactive token health checks: Every 6 hours, attempt a lightweight API call (e.g., GET /calendars/primary) per connected calendar. If it fails with 401, alert the admin immediately.
  • Implement automatic token refresh 5 minutes before expiry.
  • Track token deauth events in a calendar_tokens table with status and last_verified_at fields.
  • Dashboard should surface a Calendar Health widget showing connected calendars and their token status.

  1. Query availability for ALL pool members in parallel.
  1. Select the next rep in rotation who has availability.
  1. Book with that rep.
  1. Update the round-robin counter.

C7: Collision Detection & Resolution

Collision Types

enum CollisionType {
  HARD = 'hard',       // Same action type (e.g., both sending email) → always block
  SOFT = 'soft',       // Different action types, one is prospect-facing → warn/allow
  SEQUENCE = 'sequence', // Overlapping multi-step sequences → block
  SCHEDULING = 'scheduling' // Both trying to schedule → redirect
}

interface CollisionRule {
  action_a: string;
  action_b: string;
  collision_type: CollisionType;

Detection Logic (Redis-based, ≤10ms)

async function detectCollision(
  tenantId: string,
  recordId: string,
  agentId: string,
  actionType: string,
  cooldownWindows: CooldownConfig
): Promise<CollisionResult> {
  const key = `t:${tenantId}:actions:${recordId}`;
  const windowStart = Date.now() - (cooldownWindows[actionType] || 4 * 60 * 60 * 1000);

  // Get recent actions within cooldown window
  const recentActions = await redis.zrangebyscore(key, windowStart, '+inf');

  for (const action of recentActions) {
    const parsed = JSON.parse(action);
    if (parsed.agent_id === agentId) continue; // Same agent, not a collision

    const rule = findCollisionRule(actionType, parsed.action_type);
    if (rule && rule.default_behavior !== 'allow') {
      return {
        collision_detected: true,
        collision_type: rule.collision_type,
        colliding_agent_id: parsed.agent_id,
        colliding_request_id: parsed.request_id,
        colliding_action_type: parsed.action_type,
        colliding_timestamp: parsed.timestamp,
        recommended_behavior: rule.default_behavior,
        cooldown_remaining_ms: (parsed.timestamp + cooldownWindows[actionType]) - Date.now()
      };
    }
  }

  return { collision_detected: false };
}

C10: Communication Budget & Rate Limiting

Budget Evaluation Logic (Redis Lua Script)

-- KEYS[1] = budget sorted set key
-- ARGV[1] = current timestamp (ms)
-- ARGV[2] = window start timestamp (ms)
-- ARGV[3] = max_actions (budget limit)
-- ARGV[4] = request_id (member to add if under budget)

-- Remove expired entries
redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[2])

-- Count current entries in window
local current = redis.call('ZCARD', KEYS[1])

if current >= tonumber(ARGV[3]) then
  -- Budget exceeded
  local oldest = redis.call('ZRANGE', KEYS[1], 0, 0, 'WITHSCORES')
  local resets_at = tonumber(oldest[2]) + (tonumber(ARGV[1]) - tonumber(ARGV[2]))
  return {0, current, tonumber(ARGV[3]), resets_at}
else
  -- Under budget, add this action
  redis.call('ZADD', KEYS[1], ARGV[1], ARGV[4])
  -- Set TTL on the key
  redis.call('EXPIREAT', KEYS[1], math.ceil((tonumber(ARGV[1]) + (tonumber(ARGV[1]) - tonumber(ARGV[2]))) / 1000) + 3600)
  return {1, current + 1, tonumber(ARGV[3]), 0}
end

Budget Hierarchy Evaluation Order

  1. Per-record segment budgets (most specific) — e.g., "Enterprise accounts: max 2 emails/week"
  1. Per-agent budgets — e.g., "Agent Alice: max 500 emails/day across all records"
  1. Global budgets (least specific) — e.g., "All agents: max 4 touches per record per week"

If ANY budget is exceeded, the decision is NO_GO with reason_code: BUDGET_EXCEEDED.

function evaluatePolicies(
  context: PolicyEvaluationContext,
  policies: PolicyDefinition[]
): PolicyEvaluationResult {
  // Sort by priority (lower number = higher priority)
  const sortedPolicies = policies
    .filter(p => p.is_active)
    .sort((a, b) => a.priority - b.priority);

  const evaluatedPolicies: EvaluatedPolicy[] = [];

  for (const policy of sortedPolicies) {
    const matches = evaluateConditions(context, policy.conditions);
    evaluatedPolicies.push({
      policy_id: policy.id,
      policy_name: policy.name,
      matched: matches,
      action: matches ? policy.action : null
    });

    if (matches && policy.action.decision !== 'GO') {
      // First matching restrictive policy wins
      return {
        decision: policy.action.decision,
        reason_code: policy.action.reason_code,
        reason_text: policy.action.reason_text,
        triggering_policy: policy.id,
        all_evaluated: evaluatedPolicies
      };
    }
  }

  // No restrictive policy matched → GO
  return {
    decision: 'GO',
    reason_code: 'POLICIES_PASSED',
    reason_text: 'All policies evaluated, no restrictions apply',
    triggering_policy: null,
    all_evaluated: evaluatedPolicies
  };
}

async function computeAttribution(
  tenantId: string,
  outcomeEvent: CRMOutcomeEvent
): Promise<AttributionEvent> {
  // 1. Find the canonical record associated with this outcome
  const record = await resolveRecordFromSFDC(outcomeEvent.sfdc_record_id);

  // 2. Get all ledger entries for this record, ordered by time
  const ledgerEntries = await getLedgerEntriesForRecord(
    tenantId,
    record.canonical_record_id,
    { decision: 'GO' }  // Only count actions that were permitted
  );

  if (ledgerEntries.length === 0) {
    // Record was not coordinated
    return {
      canonical_record_id: record.canonical_record_id,
      outcome_type: outcomeEvent.type,
      was_coordinated: false,
      // ... fill in CRM fields
    };
  }

  // 3. First-touch and last-touch attribution
  const firstTouch = ledgerEntries[0];
  const lastTouch = ledgerEntries[ledgerEntries.length - 1];
  const allAgents = [...new Set(ledgerEntries.map(e => e.agent_id))];

  // 4. Calculate time-to-outcome
  const timeToOutcomeHours = (outcomeEvent.timestamp - firstTouch.created_at) / (1000 * 60 * 60);

  return {
    canonical_record_id: record.canonical_record_id,
    outcome_type: outcomeEvent.type,
    was_coordinated: true,
    first_touch_agent_id: firstTouch.agent_id,
    first_touch_request_id: firstTouch.request_id,
    last_touch_agent_id: lastTouch.agent_id,
    last_touch_request_id: lastTouch.request_id,
    all_touch_agent_ids: allAgents,
    total_coordinated_touches: ledgerEntries.length,
    pipeline_value: outcomeEvent.pipeline_value,
    time_to_outcome_hours: timeToOutcomeHours
  };
}

  1. SOC 2 Type II: Align with LeanData's existing SOC 2 controls. Implement audit logging for all admin actions. Retain admin audit logs for 24 months.
  1. Rate limiting: Per-agent API rate limits (default: 1000 requests/minute for Preflight, 100/minute for scheduling).
  1. Input validation: All API inputs validated against JSON Schema. SQL injection prevention via parameterized queries. XSS prevention on Dashboard.
  1. CORS: Dashboard SPA only. API endpoints use API key auth (no CORS needed for server-to-server).
  1. Dependency scanning: Automated vulnerability scanning in CI/CD (npm audit, Snyk, or similar).

8.3 Token & Auth Management

  • OAuth 2.0 Web Server Flow: Customer authorizes via OAuth during onboarding.
  • Token storage: Encrypted in PostgreSQL tenant_sfdc_credentials table (KMS envelope encryption).
  • Token refresh: Automatic refresh before expiry. If refresh fails, retry 3x with backoff.
  • Deauth detection: Proactive health check every 30 minutes. If 401 received, mark tenant sync as DEGRADED and alert admin.
  • Per LeanData's BookIt learnings: Calendar and SFDC token health must be visible in the Dashboard. Token deauth is a P2 alert.
